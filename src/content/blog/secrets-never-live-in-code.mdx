---
title: 'Secrets never live in code'
description: 'A secret committed to a repo is a leak waiting to happen — git history is forever. Reference secrets instead of storing them, and prefer short-lived over merely encrypted.'
pubDate: 2026-06-15
tags: ['principles', 'security', 'devsecops']
---

Of all the ways systems get breached, the one that still surprises me is how
often the credential was just — sitting in a repository. It's the most
preventable root cause there is, and it keeps happening, because committing a
secret is the path of least resistance and the cost stays invisible until it
isn't.

The principle is one line: **code references secrets; it never contains
them.** A repository describes *which* secret a service needs and *where* to
fetch it — never the secret itself.

The reason it's absolute, not a "try to avoid it," is that a secret in a repo
doesn't stay in the repo. It clones to every laptop. It fans out to forks. It
lands in CI logs, build caches, backups, and the chat message where someone
pasted the error. And Git history is permanent: committing a secret and
deleting it in the next commit removes nothing — the secret is one `git log`
away forever. The moment a credential touches version control you have to
treat it as compromised and rotate it. So don't let it touch.

What "referencing" looks like in practice:

**A secrets manager as the single source of truth.** In the systems I run,
that's Vault. Services authenticate to it at runtime and fetch what they
need; the repo and the image carry a path and an identity, not a password.

**Short-lived beats encrypted.** The instinct is to encrypt secrets at rest
and call it solved. Encryption helps, but the bigger lever is *lifetime*. A
long-lived database password — even encrypted — stays valuable indefinitely
once it leaks. A dynamically generated credential that expires in an hour is
barely worth stealing. Where the platform allows it, prefer secrets minted on
demand and expired quickly over secrets that are merely stored well.

**Injected at runtime, not baked into images.** A secret built into a
container image is baked into an immutable layer: it sits in whatever registry
the image is pushed to, in every cache that mirrors it, and in anyone's
`docker pull` — and deleting it in a later layer removes nothing, exactly like
git history. Secrets should arrive at the running process — through the
orchestrator, through the secrets manager — not in the artifact.

**Scanning as a backstop.** People slip, so pre-commit hooks and CI
secret-scanning exist to catch the mistake before it lands. The backstop
isn't the strategy; it's the seatbelt for when the strategy slips.

The mental model that ties it together: stop thinking about *protecting*
secrets and start thinking about *minimising their blast radius and
lifetime.* A credential that is short-lived, narrowly scoped, and never
written down is one you barely have to protect — there's little to steal, and
not long to steal it in.

Git forgets nothing. Make sure there's nothing to remember.
